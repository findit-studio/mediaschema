## 1. Context & Problem

`findit-proto` is a hand-written proto3-compatible serialization library (~47k LOC,
~120 message/enum types, manual wire codec; **no `.proto` files** — field numbers/wire
types are embedded as Rust constants). It mixes two very different kinds of schema:

- **Product-agnostic media primitives** — generic detection/analysis/media types
  (`Detection`, `BoundingBox`, pose/face/audio analysis, dimensions, color, …) plus the
  `mediatime` time types. Reusable outside findit.
- **findit-application schemas** — `database/` storage *records* (scene/keyframe/video/
  media, the audio subsystem) and `network/` the daemon↔desktop *protocol*. These encode
  findit's product model, not generic media concepts.

`mediaschema` is a **standalone, product-agnostic** crate: it owns only the reusable media
primitives, generated by **buffa** from clean `.proto`. The findit-application schemas are
**explicitly out of scope** — they remain in findit and will later compose `mediaschema`
primitives where useful (findit-side work, not a `mediaschema` sub-project).

This document records the cross-cutting decisions, the decomposition, and the full design
of the first sub-project (**SP0 — Foundation**).

## 2. Decisions Locked (apply to the whole effort)

1. **Source of truth = the actual Rust code** in `findit-proto` (structs, enums, oneofs,
   field semantics). Read for authoring `.proto`; `findit-proto` is **not** a dependency
   of `mediaschema` (not even dev).
2. **No byte-for-byte wire compatibility.** Requirement is *correctness for transmission*:
   round-trip fidelity. `.proto` is authored clean (idiomatic, numbered from 1).
3. **`mediaschema` is product-agnostic.** Only reusable media primitives. The findit
   `database` records and `network` daemon protocol are **excluded** and stay in findit.
4. **redb / storage-key concerns are out of scope** (they live with the excluded
   findit-application layer anyway).
5. **Required derivable capabilities:** `serde`, `arbitrary`, `quickcheck`.
6. **mediatime types are reused, not regenerated** — via buffa `extern_path` + a new
   `mediatime/buffa` feature implementing the buffa contract. mediatime is itself
   product-agnostic, so depending on it does not compromise (3).
7. **Generated code is checked in** (reviewed diffs; no `protoc`/`buf` at consumer build).
8. **Single product-neutral proto package: `media.v1`.** Not domain-mirrored (the
   `common`/`database`/`network` split was a findit module artifact). The crate flattens
   the public API to `mediaschema::<Type>` via a scoped re-export of `generated::media::v1`.
9. **findit consumer cutover is findit-side and out of these specs.** When findit adopts
   `mediaschema`, that (and any `database`/`network` redesign) happens in findit's repo,
   not as a `mediaschema` sub-project.

## 3. Decomposition

| ID | Sub-project | Scope | Sequencing |
|----|-------------|-------|------------|
| **SP0** | **Foundation** | Crate skeleton, codegen pipeline, `mediatime/buffa` feature + extern wiring, serde/arbitrary/quickcheck, round-trip harness — proven on a vertical slice | **First** (this spec) |
| SP1 | Media primitives | The product-agnostic subset of `findit-proto/common` (~detection/analysis/media types). Exact per-type membership (which `common/` types are truly product-agnostic vs. findit-infra like `id`/`file_checksum`/`watched_location`) is decided when SP1 is brainstormed. | After SP0 |
| — | findit `database`/`network` + consumer cutover | **Out of mediaschema scope.** findit-side work in findit's repo. | Not a mediaschema sub-project |

`mediaschema`'s entire scope is **SP0 + SP1**. There is no SP2/SP3 here.

## 4. SP0 — Foundation: Goal

Stand up `mediaschema` as a buffa-codegen crate with the **entire toolchain working
end-to-end** (`.proto` → checked-in generated Rust → serde/arbitrary/quickcheck →
round-trip tests), proven on a thin product-agnostic vertical slice in package `media.v1`:

- `Detection` — from `findit-proto`: `{ label: SmolStr, confidence: f32 }`
- `BoundingBox` — from `findit-proto`: `{ x: f32, y: f32, width: f32, height: f32 }`
- the `mediatime` trio via `extern_path` (`Timebase`, `TimeRange`, `Timestamp`)

SP0 de-risks the pipeline (mediatime extern integration; quickcheck bridge) before SP1
scales it.

## 5. SP0 Design

### A. Crate & repo layout

Rename the `template-rs` scaffold → `mediaschema` (package, metadata, README), neutral
description ("product-agnostic media-primitive schema, buffa-generated"). Keep the
standalone repo and branch `0.1.0`.

```
mediaschema/
  proto/media/v1/types.proto            # package media.v1: Detection, BoundingBox, TimedDetection
  proto/mediatime/v1/mediatime.proto    # package mediatime.v1; extern-mapped, NOT generated
  src/
    generated/  + mod.rs                # checked-in buffa output (relative-include stitcher)
    lib.rs                              # mod generated; pub use generated::media::v1::*;
  xtask/                                # `cargo run -p xtask -- gen` → src/generated/
  mediaschema-derive/                   # proc-macro: QuickcheckArbitrary
  tests/roundtrip.rs
```

`mediaschema` Cargo features (workspace `resolver = "2"`): `default=["std"]`;
`std`→`buffa/std`+`buffa-types/std`;
`json`→`dep:serde`+`dep:serde_json`+`buffa/json`+`buffa-types/json`+`mediatime/serde`
(feature MUST be literally `json` — buffa gates serde on `feature = "json"`);
`arbitrary`→`buffa/arbitrary`+`buffa-types/arbitrary`+`mediatime/arbitrary` (literal
`arbitrary` required by generated `cfg_attr`); `quickcheck` implies `arbitrary`,
adds `quickcheck`+`mediaschema-derive`. The `json`/`arbitrary` propagation into
`mediatime` is **load-bearing**: generated messages embed `mediatime` types via
`MessageField<T>`, and buffa's `cfg_attr` serde/arbitrary derives require
`T: Serialize`/`Arbitrary` (spec §5.C.4). Deps: `buffa`, `buffa-types`, `mediatime`
(`features=["buffa"]`). **No `findit-proto` dependency (not even dev).**

### B. Proto authoring + codegen pipeline

`.proto` authored from the **Rust structs/enums**, clean numbering from 1, single
product-neutral package **`media.v1`**. Generation is a **manual developer step**
(`cargo run -p xtask -- gen`) writing into `src/generated/` with `include_file("mod.rs")`
and an explicit `out_dir` (→ relative `include!`s; `#[allow(warnings, clippy::all)]
mod generated;`). `src/lib.rs` flattens with **named** re-exports
(`pub use generated::media::v1::{BoundingBox, Detection, TimedDetection};` — not a glob,
so buffa internals `__buffa`/`__*_JSON_ANY` stay out of the public surface) so consumers
see `mediaschema::Detection`.
**CI gate:** re-run gen; `git diff --exit-code` (mirrors buffa's `check-generated-code`).
`protoc` (default) or `buf` needed only at gen time; SP0 uses `protoc` 34.0 (verified
present); never required by consumers.

`buffa_build::Config`: `.files([proto/media/v1/types.proto]).includes([proto])
.out_dir(src/generated).include_file("mod.rs").extern_path(".mediatime.v1","::mediatime")
.generate_json(true).generate_arbitrary(true).generate_views(false)
.gate_impls_on_crate_features(true).type_attribute(".", quickcheck-cfg-derive)`.

**Critical buffa 0.6 fact:** `generate_json(true)` emits serde derives that are
*unconditional* unless `gate_impls_on_crate_features(true)` is set (buffa-codegen's
default is a hard serde dependency). With the flag on, JSON/views/text impls are
wrapped in `#[cfg(feature = "json"|"views"|"text")]` / `#[cfg_attr(feature = "json", …)]`
— the consuming crate **must name those features literally `json`/`views`/`text`**
(like `arbitrary` must be literally `arbitrary`). SP0 sets the flag on and
`generate_views(false)` (no zero-copy views — see §C), so only the `json` and
`arbitrary` (and our `quickcheck`) gates are live.

### C. mediatime ↔ buffa integration — new `mediatime/buffa` feature

The novel, highest-risk piece. buffa's documented custom-type pattern (same mechanism
`buffa-types` uses for WKTs): hand-implement `buffa::Message` and `extern_path` it.

In the **mediatime** repo, behind a new optional `buffa` feature
(`buffa = ["dep:buffa"]`, `buffa` pulled `default-features = false`), for `Timebase`,
`TimeRange`, `Timestamp`:

1. **`impl buffa::Message`** — `compute_size`/`write_to`/`merge_field`/`clear` (verified
   signatures). `Timebase` is a scalar leaf (no `SizeCache`); `TimeRange`/`Timestamp`
   nest `Timebase` → `SizeCache.reserve()/set()`/`consume_next()` pre-order pattern.
   `UnknownFields` omitted (leaf/simple — sanctioned by buffa guide; unknown tags
   `skip_field`'d). The nested `Timebase` is **always encoded** so round-trip is
   unconditional.
2. **`impl buffa::DefaultInstance`** via `buffa::__private::OnceBox` static.
3. **View aliases** at `::mediatime::__buffa::view::{Timebase,TimeRange,Timestamp}View`
   (scalar-only ⇒ `pub type FooView<'a> = Foo;`), behind `#[cfg(feature="buffa")]`.
   (Retained, but SP0 disables view generation — see below — so they are currently unused.)
4. serde/arbitrary: reuse mediatime's **existing** `serde`/`arbitrary` features.

Proto shapes (from the live mediatime structs): `Timebase{uint32 num=1; uint32 den=2;}`
(den nonzero — malformed 0 clamped to 1 on decode), `TimeRange{int64 start=1; int64 end=2;
Timebase timebase=3;}`, `Timestamp{int64 pts=1; Timebase timebase=2;}`. `mediaschema` sets
`extern_path(".mediatime.v1","::mediatime")`. SP0 spans two repos (`mediaschema` +
`mediatime`).

**ADOPTED (was a sanctioned fallback, now the decision):** SP0 sets
`.generate_views(false)`. buffa's generated view code requires extern types to provide a
`ViewEncode`/`_decode_depth`/`to_owned_from_source` view surface that the scalar-only
mediatime aliases do not implement; zero-copy views are not needed to prove the
foundation (SP0 correctness = owned round-trip). The `__buffa::view` aliases in mediatime
are retained for a possible future re-enable but are unused in SP0.

### D. serde / arbitrary / quickcheck

- **serde** — `generate_json(true)` + `gate_impls_on_crate_features(true)`: owned types
  get `#[cfg_attr(feature = "json", derive(Serialize, Deserialize))]` (proto3-canonical
  JSON). The mediaschema Cargo feature is therefore named **`json`** (buffa gates on the
  literal `json`), `json`→`dep:serde`+`dep:serde_json`+`buffa/json`+`buffa-types/json`+
  `mediatime/serde`. No unconditional serde dependency; default build has zero serde refs.
- **arbitrary** — `generate_arbitrary(true)`: structural derive, always `#[cfg_attr(feature
  = "arbitrary", …)]` (independent of the gating flag); feature literally `arbitrary` →
  `buffa/arbitrary`+`buffa-types/arbitrary`+`mediatime/arbitrary`.
- **quickcheck** — no buffa support, no off-the-shelf derive. A tiny `mediaschema-derive`
  proc-macro `QuickcheckArbitrary` expands to `impl quickcheck::Arbitrary` by sampling
  `quickcheck::Gen` bytes into `arbitrary::Unstructured` and delegating to
  `arbitrary::Arbitrary`. Injected on every generated type via
  `type_attribute(".", "#[cfg_attr(feature=\"quickcheck\", derive(::mediaschema_derive::QuickcheckArbitrary))]")`.
  Generated types are local to `mediaschema` ⇒ foreign-trait impl is orphan-rule-legal.
  `quickcheck` implies `arbitrary`. mediatime keeps its own native `quickcheck`.

### E. Correctness harness + Definition of Done

Correctness = **round-trip fidelity** + **faithful authoring** (the `.proto` is
hand-authored to mirror the source structs; human-verified during authoring — there is no
`findit-proto` coupling to assert against, by decision 1/3). Tests (`tests/roundtrip.rs`,
property-based):

- Wire round-trip, every slice type: `decode(encode(x)) == x` (owned only — views are
  disabled in SP0).
- JSON round-trip when `json`: `from_str(to_string(x)) == x`.
- mediatime extern: a `media.v1` message embedding `Timebase`/`TimeRange`/`Timestamp`
  round-trips, exercising `extern_path` + `MessageField`.

**SP0 is done when:** (1) `mediaschema` builds across feature combos (default,
no-default, std, json, arbitrary, quickcheck, all-features); (2) `cargo run -p xtask -- gen`
is `git diff`-clean in CI; (3) round-trip + JSON + extern tests pass for `Detection`,
`BoundingBox`, and `Timebase`/`TimeRange`/`Timestamp`; (4) `mediatime --features buffa`
builds and its types round-trip through a `media.v1` parent message; (5) the quickcheck
bridge is exercised by a property test.

## 6. Risks & Verification Items

- **buffa helper symbol paths** — idioms transcribed from generated `Timestamp` (scalar
  leaf) and generated `CodeGeneratorRequest` (singular nested message + `SizeCache`).
  **Re-verified against the published `buffa 0.6.0` (crates.io):** `Message`, `SizeCache`
  (`reserve`/`set`/`consume_next`), `types`/`encoding` helpers (incl. `varint_len`),
  `__private::OnceBox`, `alloc::boxed::Box`, `DecodeError::WireTypeMismatch`,
  `MessageField`, `DefaultInstance: Default + 'static`, and `buffa-build` `Config` are
  signature-identical to 0.5.x — the SP0 surface is unaffected by the 0.5→0.6 bump.
- **Extern view with nested message field** — RESOLVED: views are disabled
  (`generate_views(false)`). buffa's generated view path requires extern types to
  implement a `ViewEncode`/`_decode_depth`/`to_owned_from_source` surface the scalar-only
  mediatime aliases lack; SP0 needs no zero-copy views. Re-enabling views later would
  require implementing that surface in `mediatime`.
- **mediatime no-alloc** — the optional `buffa` feature pulls `buffa` (needs `alloc`).
  Opt-in/gated; default mediatime stays no-alloc. Recorded, not blocking.
- **Generated module nesting** — RESOLVED: confirmed tree is
  `generated::media::v1::{Detection,BoundingBox,TimedDetection}`; public API flattened via
  **named** re-exports (not a glob — keeps buffa `__buffa`/`__*_JSON_ANY` internal).

## 7. Explicitly Out of Scope

- findit `database` storage records and `network` daemon/desktop protocol schemas — these
  remain findit-application concerns, not `mediaschema`.
- redb `Key`/`Value` impls, identity types (`Id`, `FileChecksum`), uniffi FFI.
- findit consumer cutover and any `findit-proto` cross-validation — findit-side, in
  findit's repo, not a `mediaschema` sub-project.
- SP1 type volume (only the SP0 vertical slice is migrated here).
- Upstreaming a native `quickcheck` codegen mode into buffa (revisitable later).
- CI lint gates (`cargo fmt --check`, `cargo clippy -D warnings`) and SHA-pinning
  the GitHub Actions — deferred to SP1; SP0's CI is the minimal codegen-drift +
  feature-matrix gate only. NOTE: CI protoc is pinned to the local authoring
  version (34.0) — the drift gate is only deterministic if these match.
